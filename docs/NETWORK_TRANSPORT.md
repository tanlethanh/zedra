# Pairing & P2P Design

> **Status (2026-03-11):** Phase 1 PKI authentication is fully implemented.
> Sections 1–2 document the old system that has been replaced.
> Sections 5–6 describe the current implementation.
> Section 10 lists completed and pending work.

This document covers the QR pairing system, its security model, pkarr/DNS
signaling for internet-scale P2P, and the full path to relay-free connectivity.

---

## 1. Old QR Pairing System (Replaced by Phase 1)

> **This section describes the pre-Phase-1 system. It has been replaced.**
> See Section 5 for the current implementation.

### What the QR code contained

The QR payload was `zedra://<base64url(postcard(EndpointAddr))>`.

`EndpointAddr` is an iroh-base type containing:

```
EndpointAddr {
    id:    EndpointId       // 32-byte Ed25519 public key (host identity)
    addrs: BTreeSet<TransportAddr> {
        TransportAddr::Relay(RelayUrl)       // e.g. https://relay.zedra.dev
        TransportAddr::Ip(SocketAddr)        // direct UDP addresses
        ...
    }
}
```

This is purely routing information. There is **no secret**. Any device that
can read the QR code (or observe it on screen) can connect to the host.

### How auth works today (and why it fails)

The RPC layer has a `ResumeOrCreate` handshake as the first message on each
connection. The client sends an `auth_token` string. The host stores it and
uses it to resume sessions on reconnect.

The problem: the token is generated by the **client**, not the host:

```rust
// zedra-session/src/lib.rs:468-476
let auth_token = stored_auth_token.unwrap_or_else(|| {
    format!(
        "{:016x}",
        std::time::SystemTime::now()
            .duration_since(std::time::UNIX_EPOCH)
            .unwrap_or_default()
            .as_nanos()
    )
});
```

The server accepts whatever token the client sends and creates a new session
unconditionally. The token only matters for **session resumption** — it does
not prevent the initial connection. Any iroh client that knows the host's
`EndpointId` can connect.

Additionally, the named session created at startup uses a hardcoded token:

```rust
// main.rs:74
registry.create_named(&session_name, workdir.clone(), "auto").await;
```

Any client sending `auth_token: "auto"` in `SwitchSession` can hijack a named
session.

### Encoding format

The current encoding (postcard + base64url directly on `EndpointAddr`) has
two problems:

1. **No versioning envelope.** If `EndpointAddr`'s wire format changes,
   previously generated QR codes silently break.
2. **Routing info baked in.** The relay URL and direct IPs are frozen at QR
   generation time. If the host's IP changes, the QR must be regenerated.

## 2. Security Vulnerabilities in the Old Design (Fixed by Phase 1)

| Severity | Issue |
|----------|-------|
| Critical | QR code contains no secret; any scanner can connect |
| Critical | Auth token derived from nanosecond timestamp (predictable, client-generated) |
| Critical | Named sessions use hardcoded `"auto"` token |
| High     | No per-operation authorization after session bind |
| High     | Path traversal on all filesystem RPCs (`PathBuf::join` does not guard `..` or `/`) |

The transport layer (iroh QUIC + TLS 1.3) is cryptographically sound.
The application auth layer on top of it is not.

Phase 1 replaces the entire application-layer auth model with public-key
authentication. The timestamp token, hardcoded `"auto"` token, and
client-generated session identity are all eliminated.

## 3. The iroh Ticket System

### iroh-tickets 0.3.0

`iroh-tickets` is in our Cargo lock as a transitive dependency of iroh.
It provides a `Ticket` trait:

```rust
pub trait Ticket: Sized {
    const KIND: &'static str;
    fn to_bytes(&self) -> Vec<u8>;
    fn from_bytes(bytes: &[u8]) -> Result<Self, ParseError>;
    fn serialize(&self) -> String;     // KIND + base32lower(to_bytes())
    fn deserialize(s: &str) -> Result<Self, ParseError>;
}
```

Wire format uses a versioned enum (`TicketWireFormat::Variant1`) so future
iroh releases can add `Variant2` without breaking existing tickets.

We implement `Ticket` for `ZedraPairingTicket` following the same conventions.

## 4. pkarr / DNS Signaling

### Phase 1 choice: relay-free with public DNS

Phase 1 uses relay-free mode (`RelayMode::Disabled`). The host publishes its
direct IP addresses to `dns.iroh.link` via `PkarrPublisher`. The client
resolves via `PkarrResolver` and connects directly over QUIC.

**Known IP exposure:** publishing to `dns.iroh.link` makes the host's IP
visible to the pkarr server and to anyone who queries with the `EndpointId`.
The `EndpointId` is embedded in the QR so it is only known to QR scanners.
Additionally, the DNS lookup itself (the `EndpointId` query) is observable as
metadata even without reading the record body. Both are accepted trade-offs
for Phase 1.

**Future mitigation (Phase 2):** self-hosted pkarr relay with AES-GCM
encryption of the address record. The encryption key is derived from the
pairing secret (HKDF) so only authorized clients can decrypt the IP. The
pkarr relay operator sees only ciphertext. Self-hosting also eliminates the
DNS query metadata leak. See section 9.

### What PkarrPublisher publishes in relay-free mode

From iroh source:

```rust
// iroh/src/address_lookup/pkarr.rs:335-338
if data.relay_urls().next().is_some() {
    // relay url set: publish relay url only, no IPs
    data.clear_ip_addrs();
}
```

With `RelayMode::Disabled` no relay URL is set, so pkarr publishes the host's
direct UDP IP addresses. These are what the client resolves at connect time.

### Enabling pkarr (Phase 1 configuration)

**Host** (`iroh_listener.rs`):

```rust
let endpoint = iroh::Endpoint::builder()
    .relay_mode(iroh::RelayMode::Disabled)
    .secret_key(identity.iroh_secret_key().clone())
    .alpns(vec![ZEDRA_ALPN.to_vec()])
    .bind()
    .await?;

let publisher = PkarrPublisher::n0_dns()
    .build(identity.iroh_secret_key().clone());
endpoint.address_lookup().add(publisher);
```

**Client** (`zedra-session/src/lib.rs`):

```rust
let resolver = PkarrResolver::n0_dns().build();
endpoint.address_lookup().add(resolver);

// id-only addr: resolver supplies the direct IPs from pkarr
let addr = iroh::EndpointAddr::new(ticket.endpoint_id);
let conn = endpoint.connect(addr, ZEDRA_ALPN).await?;
```

## 5. Public Key Authentication Scheme (Implemented)

This replaces the old shared-secret / timestamp token model. **Fully
implemented as of 2026-03-11.** See `zedra-rpc/src/proto.rs` and
`crates/zedra-host/src/rpc_daemon.rs` for the live code.

### Design overview

```
First pairing (QR scan, one-time):
  Client scans QR → (EndpointId, handshake_key, session_id)
  iroh QUIC + TLS 1.3 established
  Client → host: RegisterClient { client_pubkey, timestamp, hmac }
  Host: verifies HMAC, marks slot consumed, adds pubkey to:
    - global authorized_clients list
    - ACL of the session the QR was generated for

Subsequent connections (reconnect / network change / app restart):
  iroh QUIC + TLS 1.3 established
  Host → client: AuthChallenge { nonce, host_signature }
  Client: verifies host_signature using stored EndpointId
  Client → host: AuthResponse { nonce, client_signature }
  Host: verifies client_signature against authorized_clients list
  Host: verifies client is in requested session's ACL
  Session proceeds
```

### Client keypair and the ClientSigner trait

The client uses a **dedicated application-layer Ed25519 keypair**, separate
from the iroh transport keypair. Reasons:

- iroh transport key rotation or protocol changes must not affect auth identity
- The signing operation is abstracted behind a trait from day one so the
  storage backend can be swapped to hardware-backed keys (Android Keystore,
  iOS Secure Enclave) without touching auth logic

```rust
// zedra-session/src/signer.rs

pub trait ClientSigner: Send + Sync {
    /// Returns the 32-byte Ed25519 public key for this client identity.
    fn pubkey(&self) -> [u8; 32];
    /// Signs `data` with the client's private key.
    fn sign(&self, data: &[u8]) -> [u8; 64];
}

/// Phase 1 implementation: file-backed key storage.
/// Future: swap for AndroidKeystoreSigner / SecureEnclaveSigner.
pub struct FileClientSigner { secret_key: ed25519_dalek::SigningKey }
```

The client generates the keypair on first launch and persists it to
`~/.config/zedra/client.key` (desktop) or app-private storage (Android).
The same keypair is reused across app restarts, network changes, and
reconnects to any host.

### ZedraPairingTicket

The QR payload. Routing info (relay URL, direct IPs) is not embedded — it is
resolved dynamically via pkarr.

```rust
// zedra-rpc/src/pairing.rs

/// QR code payload. Serialized as "zedra<BASE32LOWER>".
#[derive(Debug, Serialize, Deserialize, PartialEq, Eq)]
pub struct ZedraPairingTicket {
    /// Host's 32-byte Ed25519 public key.
    /// Used as pkarr lookup key and to verify AuthChallenge signatures.
    pub endpoint_id: iroh::EndpointId,
    /// 32-byte random key for this pairing slot.
    /// One-use: consumed by the first valid RegisterClient.
    /// Client uses this as the HMAC key to prove QR possession.
    pub handshake_key: [u8; 32],
    /// The session this QR was generated for.
    /// New client's pubkey is auto-added to this session's ACL.
    pub session_id: String,
}

impl Ticket for ZedraPairingTicket { /* postcard + base32, KIND = "zedra" */ }
```

QR URL: `zedra://<ticket.serialize()>`

### First pairing: RegisterClient

The QR path is **exclusively for first pairing**. Reconnects always use
challenge-response (section below).

```rust
// zedra-rpc/src/proto.rs

pub struct RegisterClient {
    /// Client's Ed25519 public key (32 bytes).
    pub client_pubkey: [u8; 32],
    /// Unix timestamp in seconds. Host rejects if |now - timestamp| > 60s.
    /// Prevents replay of a captured RegisterClient packet.
    pub timestamp: u64,
    /// HMAC-SHA256(handshake_key, client_pubkey || timestamp_le_bytes).
    /// Proves the sender physically scanned the QR (has handshake_key).
    pub hmac: [u8; 32],
}

pub enum RegisterClientResult {
    /// Registration accepted. Client is now authorized.
    Ok,
    /// Handshake slot was already consumed by another device.
    /// Client should prompt: "This QR has already been used.
    /// Ask the host to run `zedra qr` to generate a new one."
    HandshakeConsumed,
    /// HMAC did not verify (wrong key or tampered packet).
    InvalidHandshake,
    /// Timestamp too old or too far in the future (replay guard).
    StaleTimestamp,
}
```

Host validation steps:

1. Check `|now - timestamp| <= 60s` — reject with `StaleTimestamp`
2. Compute `expected = HMAC-SHA256(handshake_key, client_pubkey || timestamp_le)`
   and compare in constant time — reject with `InvalidHandshake`
3. Mark slot consumed atomically — reject with `HandshakeConsumed` if already
   consumed (returns this error even if HMAC is valid, to avoid oracle)
4. Add `client_pubkey` to global `authorized_clients` list
5. Add `client_pubkey` to the ACL of `session_id` from the ticket

### Handshake slot lifecycle

- **Generated**: `rand::thread_rng().fill_bytes(&mut handshake_key)` at
  `zedra start` time (one slot per session created at startup).
- **Stored**: in-memory only in `DaemonState`. A host crash before pairing
  invalidates the QR. The user re-runs `zedra qr`. This is by design — host
  crash recovery is not a case to engineer around.
- **Consumed**: atomically on the first valid `RegisterClient`. Any subsequent
  attempt with the same key returns `HandshakeConsumed`.
- **Expiry**: the slot also expires after 10 minutes regardless of use, to
  limit exposure if the QR is observed by a third party before the legitimate
  user scans it.
- **Known limitation**: if an attacker physically observes and scans the QR
  before the intended recipient, the attacker consumes the slot. The legitimate
  user sees `HandshakeConsumed` and must ask the host to regenerate. This is
  an inherent property of one-use tokens displayed on screen and is acceptable
  for the typical use case.
- **Multiple slots**: `zedra qr` can be run multiple times to generate
  independent slots for pairing multiple devices in the same session.

### Reconnect: challenge-response

On every connection that is not a first pairing (reconnect, network change,
app restart), the host issues a challenge before allowing any RPC calls.

```rust
// zedra-rpc/src/proto.rs

pub struct AuthChallenge {
    /// 32-byte random nonce, fresh per connection.
    pub nonce: [u8; 32],
    /// Ed25519 signature of the nonce by the host's iroh SecretKey.
    /// Client verifies this using the EndpointId from initial pairing,
    /// proving the challenge was issued by the real host (not a MITM).
    pub host_signature: [u8; 64],
}

pub struct AuthResponse {
    pub nonce: [u8; 32],
    /// Ed25519 signature of the nonce by the client's application keypair.
    /// Host verifies this against the stored client_pubkey.
    pub client_signature: [u8; 64],
}
```

Connection flow:

```
1. iroh QUIC + TLS 1.3 established (transport-layer endpoint auth)
2. Host generates 32-byte nonce, signs it: host_sig = sign(host_iroh_key, nonce)
3. Host → client: AuthChallenge { nonce, host_sig }
4. Client verifies host_sig using EndpointId stored from first pairing
   → if invalid: client closes connection (connecting to wrong/compromised host)
5. Client signs nonce: client_sig = signer.sign(nonce)
6. Client → host: AuthResponse { nonce, client_sig }
7. Host looks up client_pubkey in authorized_clients, verifies client_sig
   → if not in list or sig invalid: connection closed
8. Session attachment proceeds (see authorization model below)
```

The iroh TLS layer already authenticates the `EndpointId` at the transport
level. The application-layer challenge-response additionally:
- Binds the authorized application identity (registered at pairing) to the
  connection, independent of the transport layer
- Allows the transport keypair to rotate without invalidating auth
- Prepares the path for hardware-backed signing (D2)

## 6. Two-Level Authorization Model

This is analogous to SSH `authorized_keys` but with per-session granularity.

### Level 1 — Host authorized list

```
authorized_clients: Vec<ClientPubkey>
```

A pubkey in this list can connect to the host at all. It is populated by
`RegisterClient` and persisted to `~/.config/zedra/authorized_clients.json`.

Equivalent to SSH `~/.ssh/authorized_keys`.

### Level 2 — Per-session ACL

```
Session {
    id:             String,
    workdir:        PathBuf,
    acl:            Vec<ClientPubkey>,  // who may attach
    active_client:  Option<ClientPubkey>,
    pty:            ...,
}
```

A pubkey must appear in `session.acl` to attach to that session. Being in the
global authorized list is necessary but not sufficient.

**Populating the ACL:**
- QR is always generated in the context of a specific session/workspace:
  `zedra qr --session <id>` (or `zedra qr` for the default session)
- When `RegisterClient` succeeds, the new `client_pubkey` is automatically
  added to `session.acl` for the session named in the ticket
- No manual `add-client` CLI step is needed for the common case
- A client may appear in the ACL of multiple sessions simultaneously
  (e.g. authorized for both a `~/work` and `~/personal` workspace session)

### Session ownership: exclusive single-client per session

Only one client may be actively attached to a session at a time. This prevents
split-brain on the PTY (two clients writing to the same terminal).

**Taking over a session:**

1. New device pairs via QR (pubkey auto-added to session ACL)
2. The current active client is still attached
3. New client attempts to attach → host rejects with `SessionOccupied`
4. To transfer ownership, the active client must first be disconnected:
   - **Voluntarily**: active client disconnects (app background, `zedra detach`)
   - **Forced by host**: `zedra detach --session-id <id>` sends
     `SessionTakenOver` close reason to the active client before dropping
     the connection, so the client can show a UI message rather than just
     seeing a dropped connection
5. After the session is unoccupied, the new client may attach

```rust
// Close reasons sent to active client on forced detach
pub enum SessionCloseReason {
    /// Host operator ran `zedra detach --session-id <id>`
    SessionTakenOver,
    /// Host process is shutting down
    HostShutdown,
}
```

### CLI surface for authorization management

```bash
zedra sessions                   # list sessions with active client and ACL
zedra clients                    # list all globally authorized pubkeys
zedra revoke <pubkey_hex>        # remove from authorized_clients + all ACLs
zedra detach --session-id <id>   # force-disconnect active client of a session
zedra session acl <id>           # list pubkeys in a session's ACL
zedra session acl-remove <id> <pubkey_hex>  # remove from one session's ACL only
```

## 7. P2P Connectivity Tiers

### Three tiers

```
Tier 1: Relay-assisted
  QR → pkarr (relay URL) → connect via relay → hole punch → direct P2P

Tier 2: Relay-free via pkarr direct IPs  ← Phase 1 choice
  QR → pkarr (direct IPs) → connect directly (no relay at any point)
  Host: RelayMode::Disabled; pkarr publishes IPs

Tier 3: mDNS (LAN only, zero infrastructure)
  No QR needed for discovery → mDNS → connect to LAN IP directly
  Auth flow unchanged: QR still needed for first pairing
```

### IP exposure comparison

| | Relay-assisted | Relay-free (Phase 1) |
|-|---------------|---------------------|
| Host IP in pkarr record | No (relay URL only) | Yes (direct IPs) |
| Host IP visible to relay operator | Yes (connection metadata) | N/A |
| Host IP visible to dns.iroh.link | No | Yes |
| EndpointId lookup observable by dns.iroh.link | Yes | Yes |
| Works behind NAT / CGNAT | Yes | Reachable IP only |
| Latency | relay + hole-punch overhead | Direct UDP |

Phase 1 accepts IP exposure in exchange for zero relay infrastructure and
lower latency. Phase 4 mitigates it via encrypted pkarr records on a
self-hosted relay.

### Tier 3: LAN mDNS

Deferred. See section 13 (Future Improvements).

## 9. Connection Lifecycle & Reconnect Strategy

### Connection states

```
┌─────────────────────────────────────────────────────────────────┐
│ Connected                                                       │
│  • ping every 2s (foreground only), RTT tracked for badge      │
│  • 5 consecutive missed pongs → Reconnecting                    │
│  • QUIC error / connection closed → Reconnecting               │
│  • HostShutdown received → show "Host disconnected" briefly     │
│    → auto-transition to Reconnecting after 2s                  │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ Reconnecting (attempt N / 10)                                   │
│  • invalidate pkarr cache, fresh re-resolve on each attempt     │
│  • on connect: full challenge-response → Connected              │
│  • backoff per attempt:                                         │
│    1s, 2s, 4s, 8s, 16s, 30s, 30s, 30s, 30s, 30s               │
│    (total window before giving up: ~3 minutes)                  │
│  • after 10 failed attempts → Host Unreachable                  │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│ Host Unreachable                                                │
│  • UI: "Host unreachable after 10 attempts"                    │
│  • "Retry" button → reset attempt counter → Reconnecting        │
│  • Long press workspace → "Remove" → delete stored pairing     │
└─────────────────────────────────────────────────────────────────┘
```

### Ping / pong wire protocol

Added to `zedra-rpc/src/proto.rs`:

```rust
pub struct Ping {
    /// Client's current Unix timestamp in milliseconds.
    /// Echoed back in Pong so the sender can compute RTT = now - timestamp_ms.
    pub timestamp_ms: u64,
}

pub struct Pong {
    pub timestamp_ms: u64,
}
```

- Host sends `Ping` every 2 seconds while a client is attached.
- Client responds with `Pong` echoing `timestamp_ms`.
- RTT = `now_ms - pong.timestamp_ms`. Displayed in the transport badge.
- 5 consecutive unanswered pings → host-side detection of dead client.
- Client-side: if no `Ping` received for 5 intervals (10s) → connection
  considered dead → Reconnecting.

### Background / foreground handling

On Android and iOS the OS suspends network activity when the app is
backgrounded (Doze mode, background app refresh limits). Missed pings during
background state must **not** count toward the 5-miss threshold, otherwise
every app switch would trigger a spurious reconnect.

Rules:
- Ping counter is only incremented while the app is in the foreground.
- On returning to foreground: if the QUIC connection is still alive, resume
  ping/pong normally. If the OS has already surfaced a connection error,
  enter Reconnecting immediately (do not wait for 5 missed pings).
- When transitioning from foreground → background, reset the miss counter.

### Host shutdown: graceful signal

When the host process exits cleanly (`zedra stop`, SIGTERM), it closes the
QUIC connection with `SessionCloseReason::HostShutdown` before exiting. The
client receives this close reason immediately and can:

- Show a brief "Host disconnected" message (not "Connection lost") for better UX.
- Auto-transition to Reconnecting after 2s, using the same 10-attempt flow.
  The `HostShutdown` signal does not mean "do not reconnect" — the daemon may
  restart. It only improves the UI message relative to a silent crash.

### pkarr cache invalidation on reconnect

Each reconnect attempt must perform a **fresh pkarr resolve**, not reuse the
cached `EndpointAddr` from the previous successful connection. This ensures
that if the host's IP changed (network interface update, DHCP renewal), the
client picks up the new IP rather than repeatedly failing to reach the old one.

The iroh `PkarrResolver` caches records. To force re-resolution, the client
must either:
- Reconstruct the resolver per reconnect attempt (simple, slight overhead), or
- Call the resolver's cache invalidation API if iroh exposes one.

Either way, the `EndpointId` stays stable (it is the host's persistent identity
key), so only the IP lookup result changes.

## 11. QR Code Size Budget

Error correction level L (7% recovery).

| Payload | Raw bytes | Base32 chars | QR version |
|---------|-----------|-------------|------------|
| EndpointId + relay URL + 2 IPs (pre-Phase 1) | ~95 | ~153 | v5–6 |
| EndpointId + handshake_key + session_id (Phase 1, current) | ~72 | ~116 | v4 |
| EndpointAddr (relay URL + 2 LAN IPs) + handshake_key + session_id (future) | ~112 | ~180 | v6 |

Phase 1 QR is smaller than the pre-Phase 1 one despite adding a 32-byte key and
session ID, because the relay URL and direct IPs are no longer embedded.

The future embedded-addr variant adds back the routing hints (~40 bytes) for
local-network connects. It is still a low-density QR at version 6, scanning
fast on any modern device.

## 13. Future Improvements

### Embed routing hints in ZedraPairingTicket for local-network connects

**Problem:** `ZedraPairingTicket` currently stores only `endpoint_id: iroh::PublicKey`.
On first connect the client constructs an id-only `EndpointAddr` and depends on
pkarr to supply routing info. This has two failure modes:

1. **No internet / local network:** pkarr is unreachable; the client cannot
   resolve the host at all, even though it is sitting on the same LAN.
2. **Relay mode enabled:** when the host has a home relay, iroh's `PkarrPublisher`
   publishes **only the relay URL** and suppresses direct IPs
   (see `iroh/src/address_lookup/pkarr.rs`):
   ```rust
   if data.relay_urls().next().is_some() {
       data.clear_ip_addrs();  // relay set → don't publish IPs
   }
   ```
   So even if the client is on the same LAN, pkarr resolution sends it to the
   relay. The local-network shortcut is never used.

**Fix:** replace `endpoint_id: iroh::PublicKey` with `addr: iroh::EndpointAddr`
in `ZedraPairingTicket`. `EndpointAddr` already holds both relay URL and direct
`SocketAddr`s in a single `BTreeSet<TransportAddr>`:

```rust
// iroh-base-0.96/src/endpoint_addr.rs
pub struct EndpointAddr {
    pub id:    EndpointId,
    pub addrs: BTreeSet<TransportAddr>,  // TransportAddr::Relay | TransportAddr::Ip
}
```

The host already has the full populated `EndpointAddr` at QR-generation time
(`endpoint.addr()` in `qr.rs`). Embedding it means:

- **LAN connect:** iroh tries the embedded local IP (`192.168.x.x`) first;
  relay and pkarr are never involved.
- **Remote connect:** relay URL is present as a hint; pkarr is still used to
  pick up fresher IPs but the relay hint gives an immediate fallback.
- **Offline / airgapped:** works with no DNS at all as long as the IP in the
  QR is still valid.

**Filter before embedding.** The host should strip addrs that are unlikely to
be valid from the client's perspective:

| Keep | Drop |
|------|------|
| `192.168.x.x`, `10.x.x.x`, `172.16-31.x.x` (LAN) | `127.x.x.x`, `::1` (loopback) |
| `relay_url` (always useful) | Ephemeral port-mapped public IPs (may not be stable) |

**QR size:** `EndpointAddr` with one relay URL + two LAN IPs adds ~40 bytes over
the current pubkey-only ticket. See updated size budget in section 11.

**Protocol note:** the `endpoint_id` field used for host-signature verification
becomes `addr.id` — no semantic change.

**Migration:** the change is a breaking wire format update to `ZedraPairingTicket`.
Existing persisted workspace addrs (in `PersistedWorkspace.endpoint_addr`) are
already full `EndpointAddr` blobs and are unaffected; only the QR payload changes.

### LAN mDNS discovery

Deferred from the active implementation plan. Revisit if there is a concrete
need for fully offline LAN usage or zero-infrastructure local discovery.

**What it adds over Phase 1:**
- Auto-discovery of nearby hosts in the home screen UI — paired clients see
  "Nearby: my-macbook" without a pkarr round-trip
- Faster LAN reconnect — no DNS lookup latency
- Fully offline LAN — works with no internet access

**Why it is deferred:**
Phase 1 with `RelayMode::Disabled` causes pkarr to publish all local IPs
alongside the public IP. A pkarr-resolved connection from the same LAN will
typically resolve to the local IP directly, so mDNS routing benefit is
marginal. The auth flow still requires a QR scan for first pairing regardless,
which limits the "zero infrastructure" appeal.

**Implementation notes (when needed):**

```toml
iroh = { version = "0.96", features = ["address-lookup-mdns"] }
```

```rust
// Both host and client
let mdns = MdnsAddressLookup::builder()
    .service_name("zedra")
    .build(endpoint.id())?;
endpoint.address_lookup().add(mdns);

// Client: subscribe to discovered hosts for home screen "nearby" list
let mut events = mdns.subscribe().await;
// DiscoveryEvent::Discovered { endpoint_info } → display in UI
```

Android requires:
```xml
<uses-permission android:name="android.permission.CHANGE_WIFI_MULTICAST_STATE"/>
```

The Phase 1 auth flow (QR + challenge-response) is unchanged when using mDNS
for routing — mDNS only replaces the pkarr lookup step.
